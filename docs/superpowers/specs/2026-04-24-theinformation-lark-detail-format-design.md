# The Information 飞书明细文档层级优化设计

## 目标

仅优化飞书明细文档的可读性，让多篇文章之间的边界更明显、单篇文章内部层级更清晰。

本次改动只影响自动模式发布到飞书的明细文档 Markdown，不修改本地生成的 `json`、`txt`、`html` 输出格式，也不改动任何文章正文内容。

## 范围

- 影响对象：飞书明细文档
- 不影响对象：本地 `theinformation-brief.*` 文件、飞书索引文档
- 内容约束：只调整标题层级与编号形式，不改写文案、不增删信息

## 目标结构

每篇文章在飞书明细文档中使用独立的顶层分组标题：

```md
## 文章一
## 文章二
```

每篇文章内部统一采用中文层级：

```md
一、文章信息
1. 栏目：...
2. 作者：...
3. 发布时间：...
4. 原文链接：...

二、核心观点
...

三、关键数据与事实
1. ...
2. ...
3. ...
```

## 映射规则

- 文章级别：
  - 第一篇显示为 `文章一`
  - 第二篇显示为 `文章二`
  - 依次类推
- 文章标题：
  - 保留现有英文标题
  - 若存在中文标题翻译，则紧跟在英文标题下方显示
- 文章信息区：
  - 统一归入 `一、文章信息`
  - 其中栏目、作者、发布时间、原文链接按 `1. 2. 3. 4.` 编号
- 正文主段落：
  - 将现有大块结构映射为 `二、三、四...`
  - 各段内已有条目内容保持原文，仅在必要时补成 `1. 2. 3.` 有序列表

## 实现方式

采用“飞书发布前重排 Markdown”的方案。

- 保留当前本地 brief 生成链路不变
- 在飞书发布载荷生成阶段新增一层“飞书专用 Markdown 整理”
- 由该整理层把现有 brief 内容重组成更适合飞书阅读的结构
- 发布脚本继续使用整理后的 Markdown 覆盖/创建飞书明细文档

## 方案取舍

选择该方案的原因：

- 改动面最小，只作用于飞书发布链路
- 不会影响当前本地产物和已有页面样式
- 失败面清晰，验证成本低
- 后续若要继续微调飞书排版，可以集中在同一层处理

未选择的方案：

- 不直接修改本地 brief 渲染器，避免一处排版需求影响全部输出渠道
- 不采用“先发文再二次编辑”的方式，避免流程脆弱和接口调用变复杂

## 验证方式

- 单元测试验证飞书 Markdown 生成结果包含：
  - `文章一 / 文章二`
  - `一、文章信息`
  - `1. 栏目`
  - `二、核心观点`
- 真实发布后验证：
  - 同一天重跑仍然更新同一篇飞书明细
  - 索引文档逻辑不受影响
  - 本地输出文件内容不变
